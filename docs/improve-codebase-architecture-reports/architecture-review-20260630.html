1. Deepen the social button popup machinery into one module

Strong ports & adapters
packages/social/src/LikeButton.tsx
packages/social/src/ShareButton.tsx
packages/social/src/SubscribeButton.tsx

Before

LikeButton

SubscribeButton

localStorage · state · label

≈85% identical implementation, 3× tests to maintain

After

useSocialAction(config)
popup open · token generation
postMessage listener · poll fallback
abort on unmount · onSuccess dispatch
LikeButton
thin adapter
ShareButton
thin adapter
SubscribeButton
thin adapter

All tests hit the deep module · buttons are 20-line adapters

Problem. Three buttons each contain the full popup/polling/postMessage machinery — 140–156 lines each, 85% identical. The differences (action type, localStorage vs timeout reset) are tiny but buried inside the duplication. Every bug fix and every new feature touches three files.

Solution. Extract a useSocialAction(config) hook — the deep module — that owns popup lifecycle, token generation, postMessage, polling fallback, abort, and onSuccess dispatch. Each button becomes a thin adapter: a 20-line wrapper that configures useSocialAction with its action type and persistence strategy.

Wins

  • → locality: one place to fix popup/polling bugs, fix once for all three buttons
  • → leverage: one test suite exercises the shared machinery; adapter tests need only assert prop-passing
  • → interface shrinks — persistence and reset strategies are config, not reimplementation
  • → delete 300 lines of duplicated implementation

2. Make the PDS resolution cache clearable across a seam

Strong in-process
packages/core/src/resolve.ts
packages/core/src/fetch.ts

Before — three opaque caches

flowchart TB
  subgraph resolve ["resolve.ts (module-private)"]
    HC["handleCache: Map"]
    PC["pdsCache: Map"]
  end
  subgraph fetch ["fetch.ts"]
    PUC["publicationUriCache: Map\n(exported _clear fn)"]
  end
  resolve -. "no seam\nno clear" .-> T["Tests\n❌ unique-DID workaround"]
  fetch -- "_clearPublicationUriCache()" --> T
  classDef leak stroke:#dc2626,stroke-width:2px;
  class HC,PC leak

After — one seam, one clear

flowchart TB
  subgraph cache ["ResolutionCache (deep module)"]
    I["handle → DID\nDID → pdsUrl\npublicationUrl → AT URI"]
  end
  cache -- "clear()" --> T["Tests\n✓ isolated per test"]
  F["fetchSite / fetchArticle"] --> cache
  classDef deep fill:#0f172a,color:#f8fafc,stroke:#0f172a;
  class cache deep

Problem. Three independent caches (handleCache, pdsCache, publicationUriCache) live as module-level globals. Two are private to resolve.ts — tests cannot clear them. The test suite works around this by assigning a unique DID to every test, a fragile contract that will silently break if a test reuses a value.

Solution. Consolidate all three caches into a single ResolutionCache module with a clean seam: a clear() method exported for tests and a get/set for the three resolution steps. fetchSite and fetchArticle accept an optional cache instance — defaulting to a shared singleton so production behaviour is unchanged.

Wins

  • → locality: all caching logic in one module — TTL or invalidation lands in one place
  • → leverage: tests cache.clear() in beforeEach — the unique-DID workaround disappears
  • → seam becomes real — production uses the singleton; tests can inject a fresh instance

3. Collapse duplicated raw AT Protocol interfaces

Worth exploring in-process

After — one source of truth

raw-types.ts (new)
AT Protocol raw shapes
ScribeManifest
RawPublication
RawDocument
fetch.ts
imports
list.ts
imports

Lexicon change touches one file

Problem. ScribeManifest, RawPublication, and RawDocument are defined independently in both fetch.ts and list.ts — but with different field sets. RawDocument.updatedAt is required in one, optional in the other; canonicalUrl and contributors exist only in fetch.ts. When the AT Protocol lexicon changes, both files must be updated in sync.

Solution. Extract a raw-types.ts module — a single source of truth for AT Protocol wire shapes. Both fetch.ts and list.ts import from it. The seam is the AT Protocol lexicon itself.

Wins

  • → locality: lexicon change touches one module
  • → interface shrinks — fetch.ts and list.ts lose their private type definitions

Problem. feed.ts and sitemap.ts each define a private function that builds the same article URL from (baseUrl, urlPrefix, groupSlug, articleSlug) — identical logic, different names. If URL routing rules change, both must be updated.

Solution. Move to utils.ts as buildArticleUrl(baseUrl, urlPrefix, groupSlug, articleSlug), exported and tested once. Both feed.ts and sitemap.ts import it.

Wins

  • → locality: URL routing rule change touches one module
  • → leverage: one test, used by both feed and sitemap generation

5. Deepen createScribeSite by caching its site fetch internally

Speculative in-process
packages/next/src/create-scribe-site.ts

Before — 5 fetches, relies on global cache

sequenceDiagram
  participant N as Next.js
  participant C as createScribeSite
  participant F as fetchSite
  N->>C: generateGroupParams()
  C->>F: fetchSite() #1
  N->>C: generateArticleParams()
  C->>F: fetchSite() #2
  N->>C: generateGroupArticleParams()
  C->>F: fetchSite() #3
  N->>C: generateGroupMetadata()
  C->>F: fetchSite() #4
  N->>C: generateArticleMetadata()
  C->>F: fetchSite() #5

After — one fetch, explicit promise

sequenceDiagram
  participant N as Next.js
  participant C as createScribeSite
  participant F as fetchSite
  N->>C: generateGroupParams()
  C->>F: fetchSite() — cached promise
  F-->>C: site
  C-->>N: params
  N->>C: generateArticleParams()
  C-->>N: params (no fetch)
  N->>C: generateGroupMetadata()
  C-->>N: metadata (no fetch)

Problem. Five of the six methods returned by createScribeSite call fetchSite() independently. At build time, Next.js may call them in parallel — the global publicationUriCache in core may de-duplicate the network call, but this is an invisible implementation detail, not a guarantee at this module's interface.

Solution. Cache the site as a Promise<Site> inside createScribeSite's closure — fetched once on first call to any method, shared by all subsequent calls. The cache is scoped to the factory instance, not to the global module.

Note: This is speculative — the global cache in core may already make the problem negligible in practice. Worthwhile only if build-time profiling shows redundant PDS fetches.

Wins

  • → depth increases — the factory's interface is already small; this makes its implementation more self-contained
  • → locality: caching is an implementation detail of the factory, not an implicit dependency on the global cache

Top recommendation

Start with candidate 1 — deepen the social button module

The three social buttons share a non-trivial popup/polling/postMessage machinery that is currently duplicated in full. Every bug fix costs three edits; every new feature (e.g. the planned display counter) costs three implementations. Extracting useSocialAction as a deep module concentrates all that logic — and all the tests — behind one interface. The buttons themselves become thin adapters: easy to audit, easy to extend, nearly impossible to diverge.
